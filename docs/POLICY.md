# 方針書

開発方針を「前提条件 → 大方針 → 中方針 → 小方針」の順で管理する。
本プロジェクトは**設計先行（ウォーターフォール）**で進め、設計を固めてから実装する。

---

## 前提条件（P）

各方針が依存する前提。前提が崩れたら依存する方針を再評価する。

| P# | 内容 | 確度 |
|----|------|------|
| P1 | 一人開発・ポートフォリオ目的 | 確定 |
| P2 | 開発環境：WSL2 + Go + Node | 確定 |
| P3 | アピール対象＝バックエンド寄りの領域 | 高 |
| P4 | ホスティングは無料枠中心、まず PaaS → 将来クラウド | 中（後続で確定） |
| P5 | パーツ仕様は現時点未定（後続の中/小方針で定義） | 要確認 |

---

## 大方針（確定）

| 大# | 内容 | 依存 |
|-----|------|------|
| 大1 | ジャンル：マルチプレイ／通信ゲーム（ロボット組立オートバトラー） | P1, P3 |
| 大2 | プラットフォーム：Web ブラウザ | P1, P3 |
| 大3 | アピール軸：サーバー／通信／インフラ ＋ 開発プロセス／設計 | P3 |
| 大4 | 進め方：ウォーターフォール（設計先行） | P1 |
| 大5 | 対戦方式：ハイブリッド（非同期＝指名チャレンジ制 ＋ 同期＝ライブ観戦／「決定論シミュレーション＋リプレイ」を共通基盤とする） | P1 |
| 大6 | 行動アルゴリズム：ルールベースを土台 → ビジュアルプログラミングを最終目標 | P5 |
| 大7 | 設計の柱：アルゴリズムの組み方でパーツ差を覆せる | 大6 |
| 大8 | 技術スタック：サーバー Go + WebSocket／主 DB MySQL（将来 Redis）／フロント TypeScript + Canvas／ホスティング PaaS → クラウド | P2, P3, P4 |

### コンセプト補足
ロボットのパーツと行動アルゴリズムを組み立て、ロボット同士を自動で戦わせて観戦する対戦ゲーム。
プレイヤーは戦闘中は操作せず、事前に組んだ構成と行動ロジックだけで勝敗が決まる。
