# 方針書

開発方針を「前提条件 → 大方針 → 中方針 → 小方針」の順で管理する。
本プロジェクトは**設計先行（ウォーターフォール）**で進め、設計を固めてから実装する。

## このプロジェクトの位置づけ・目的
- **目的**：**パーツ選択と行動アルゴリズムを自由に組む**ことを追求したゲームシステムを確立する。**2D でも面白い**ゲームを目指す。
- **次回作のたたき台**：本 2D 版で確立する戦闘システム（行動アルゴリズム／パーツ／戦闘ルール）は、次に **Unity で作る 3D 版 RoboGame のたたき台**とする。3D 化より先に、ゲームの根幹を 2D で作り切る。

### コンセプト補足
ロボットのパーツと行動アルゴリズムを組み立て、ロボット同士を自動で戦わせて観戦する対戦ゲーム。
プレイヤーは戦闘中は操作せず、事前に組んだ構成と行動ロジックだけで勝敗が決まる。

## 中方針

### 中1 パーツ仕様（確定）

- **素体（ベース）**：基礎HP・基礎重量・スロット数・総バッテリー（電力上限）・基礎移動速度を提供。**個性**（長所短所：HP↔重量のトレードオフ等）を持つ。装備なしでは移動のみ可能。最初はバランス型1種からテスト。
- **装備カテゴリ**：
  - 移動系（ダッシュ）：通常移動に加え、高速移動（短時間・消費電力・クールダウン）を追加
  - 攻撃系（攻撃ユニット＋武器）：武器が攻撃力／射程／攻撃パターン／発熱量を規定
  - 防御系（装甲＝シールド値／防御ユニット＝能動的な防御行動）
  - 電力系（電力割り振り）：**任意・上級**。無くても戦える
- **耐久の二層化**：HP（素体本体・**0で完全破壊**）の上に**シールド**（装甲が付与・**回復なしの使い捨て**）。ダメージは**シールド→HPの順**。
- **熱システム（銃系武器）**：発射で熱が蓄積。熱に**比例してバッテリー消費が増加**。**オーバーヒート＝武器ロック＋追加ペナルティ**（大電力消費・速度低下等）。冷却はパーツ非依存で、**発射ペース管理＝アルゴリズムの役割**（下手な組み方は自滅し再設計を促す）。
- **制約**：スロット数（素体依存）／総重量（＝素体＋全パーツ。重いほど移動速度低下・バッテリー負荷増）／エネルギー（総バッテリー容量 vs 消費電力）。
- **エネルギー超過の扱い**：基礎＝案A（超過は装備不可）→ 拡張＝案B（超過許容＋戦闘中の電力不足を電力割り振りプログラムで救済）。
- **パーツ共通属性**：重量・消費電力・スロット消費 ＋ カテゴリ固有効果。
- **柱との関係**：熱管理・シールド/HPの使い分け・電力管理という**管理スキルをアルゴリズム側に乗せ**、パーツ差をアルゴリズムで覆せるようにする。
- **段階**：基礎（バランス型素体＋最小装備で骨格検証）→ 拡張（エネルギー/案B・装甲・防御・移動系・複数素体&個性・電力割り振り）。

### 中2 シミュレーション仕様（確定）

- **空間**：2D 見下ろし平面 (X, Y)。決定論は整数ベクトルで担保し、距離は二乗値の比較で扱う（平方根を使わない）。
- **時間**：離散ティック進行（ティックレート等の数値は小方針）。
- **移動**：
  - 通常移動＝素体の基本能力。**速度＝基礎速度 −（総重量ペナルティ）**。常時可能。
  - ダッシュ＝移動系パーツで追加。**短時間の高速移動＋消費電力＋クールダウン**。アルゴリズムが発動タイミングを決める。
- **行動の同時実行**：1 ティックに、**資源（電力・熱）の許す範囲で複数行動が可能**（例：移動しながら攻撃）。電力・熱のやりくりがアルゴリズムの腕。
- **攻撃・ダメージ解決**：武器の攻撃力／射程／攻撃パターンに従う。ダメージは**シールド→HP の順**。HP 0 で完全破壊。
- **熱**：銃系は発射で熱蓄積 → 熱に比例してバッテリー消費増 → オーバーヒートで武器ロック＋追加ペナルティ。冷却は発射ペース管理（アルゴリズム）。
- **バッテリー**：各行動・パーツが電力を消費。基礎＝案A（超過は装備不可）→ 拡張＝案B（戦闘中の電力不足を電力割り振りで救済）。
- **解決順序**：両者の行動を**ティック開始時点のスナップショットから同時解決**（先手有利を出さない・決定論的）。
- **終了条件**：いずれかの HP が 0 で決着。規定ティック数で**タイムアウト → 残 HP（次いで残シールド）が多い方の勝ち・同値は引き分け**。
- **段階2改訂**：hitscan攻撃を**発射体（弾速・回避可）に置換**し、**遮蔽物（カバー）**と**マップ拡大（対称初期配置＋前方カバー）**を追加する。詳細は BasicDesign §9「2-G 戦闘モデルの進化」。

### 中3 行動アルゴリズム表現（確定）

- **構造：チャネル分割 ＋ 優先順位ルール**。行動を独立した複数チャネルに分け、各チャネルが優先順位つきルールリストを持ち、毎ティック**同時に評価**される（1ティック複数行動＝中2案Y と整合）。
  - 移動チャネル／武器チャネル／（拡張）特殊チャネル
- **ルール** ＝ 条件（複数 AND 可）→ 行動。各チャネル内で上から評価し、最初に合致した行動を実行。どれも合致しなければ**デフォルト動作**（移動＝接近／武器＝撃たない）。
- **条件パレット（初期）**：敵との距離（射程内/射程外/近い/遠い）、自HP割合（高/低）、自シールド（あり/なし）、自熱（低/高/オーバーヒート間近）、自バッテリー残量（高/低）、ダッシュ使用可否。
- **行動パレット（初期）**：
  - 移動：接近／後退／距離維持（カイティング）／ダッシュ接近／ダッシュ離脱／停止
  - 武器：攻撃する／撃たない（冷却）
  - 特殊（拡張）：防御行動／電力割り振り切替
- **連動**：使える行動は装備パーツが決める（中1連動）。
- **最終目標**：この優先順位ルールを**ビジュアルプログラミング**で組めるようにする（大6）。土台はルールベース。

### 中4 データモデル（確定）

**ビルド系（ロボットの定義）**
- **Chassis（素体）**：name／基礎HP／基礎重量／スロット数／バッテリー容量／基礎速度／個性
- **Part（パーツ）**：name／カテゴリ(武器/装甲/移動/電力)／重量／消費電力／スロット消費 ＋ カテゴリ固有
  - 武器：攻撃力・射程・攻撃パターン・発熱量／装甲：シールド値／移動(ダッシュ)：ダッシュ距離・速度・クールダウン・消費電力
- **Ruleset（行動）**：チャネル(移動/武器/特殊)ごとの Rule[]。**Rule** ＝ 条件[]（AND）→ 行動
- **Build**：Chassis ＋ Part[] ＋ Ruleset。導出値（総重量・使用スロット・電力収支・総シールド・実効速度）で制約検証

**保存・対戦系**
- **Robot（名簿の1体）**：id／owner／Build
- **Battle**：id／挑戦者(Robot)／相手(Robot)／winner／reason／Replay
- **Replay**：両者の Build／Frame[]／winner／reason（どんな構成が勝ったか再現可能）
- **Frame**：tick／各ロボの状態[2]＝{座標(x,y)・HP・シールド・熱・バッテリー・ダッシュCD・オーバーヒート}／Event[]
- **Event**：種別(攻撃/ダッシュ/オーバーヒート/破壊 等)／source／target／量
- **Inbox**：owner ごとの対戦結果一覧（防衛側視点）

Ruleset は入れ子 JSON（チャネル→ルール→条件/行動）で表現する。

### 中5 通信・API方式（確定）

**非同期（REST）＝主軸**
- `POST /api/robots`：ビルド登録（{owner, Build}。制約検証して保存）
- `GET /api/robots`：名簿一覧
- `POST /api/challenge`：{challenger_id, opponent_id} → サーバーが sim → Battle（Replay 含む）を返却＋相手の inbox へ記録
- `GET /api/inbox?owner=`：受信箱（防衛側の結果一覧）
- `GET /api/battles/{id}`（任意）：対戦詳細・リプレイ取得

**同期（WebSocket）＝拡張段階**
- `WS /ws/...`：2 人が同じ戦闘をライブ同時観戦。サーバーが決定論 sim のフレームを配信、ルーム管理・再接続。基礎は REST のみで、これは後段。

**型共有方針**：初期は JSON（Go と TS が共通スキーマ＝基本設計書で定義 に従う）→ 拡張で Protocol Buffers による型自動生成。

**その他**：CORS は開発時プロキシで同一オリジン化・本番はオリジン限定。エラー応答は `{error: メッセージ}` ＋ 適切な HTTP ステータス。

### 中6 画面構成・遷移（確定）

**画面**
1. **ビルド作成**（2ステップ）
   - ①組立：素体選択＋パーツ装着。制約をリアルタイム表示（スロット・総重量・電力収支・総シールド・実効速度）。超過時は警告（基礎=案Aで装備不可）
   - ②行動アルゴリズム編集：チャネル(移動/武器/特殊)ごとに「条件→行動」ルールを優先順位で編集（ルールベースUI、将来ビジュアル）
2. **名簿・相手選択**：登録済みロボ一覧から相手を選んで挑戦
3. **観戦**：Canvas でリプレイ再生（座標・HP・シールド・熱・バッテリー・イベント表示）
4. **受信箱**：挑まれた結果一覧＋「観戦」で再生
- 軽い識別：owner 名（認証は後段）

**遷移**：ビルド作成(組立→アルゴリズム編集) → 登録(名簿入り) → 相手選択 → 挑戦 → 観戦／受信箱 → 観戦

### 中7 永続化（確定）

- **段階**：基礎は**インメモリ**（再起動で消える）で骨格検証 → 確立後に **MySQL** で永続化。
- **MySQL スキーマ方針**：入れ子データ（Build・Ruleset・Replay）は **JSON カラム**で保存（過度に正規化しない）。
  - `robots`（id, owner, name, build JSON, created_at）
  - `battles`（id, challenger_robot_id, opponent_robot_id, winner, reason, replay JSON, created_at）
  - **受信箱**は専用テーブルを作らず、`battles` を相手(opponent)の owner で絞るクエリで導出。
- **Redis**：複数インスタンス化／同期モードのスケール時に導入（大8）。基礎では不要。

### 中9 セキュリティ・インフラ（確定）

- **サーバー権威・チート対策**：戦闘はサーバー側の決定論 sim で計算、クライアントは観戦のみ。リプレイはサーバー生成で改ざん不可。
- **入力検証**：ビルド登録時にサーバーで検証（ステータス正値・制約＝スロット/重量/電力収支、Ruleset の条件/行動が既知の妥当なものか、チャネル構造の妥当性）。
- **認証**：基礎は軽い owner 名のみ（認証なし）。段階3以降で簡易認証を検討。
- **CORS**：開発はプロキシで同一オリジン化、本番はオリジン限定。
- **レート制限・濫用対策**：後段で任意検討。
- **インフラ/デプロイ**：フロント＝静的ホスティング、サーバー＝常時接続可能な PaaS（WebSocket 前提でサーバーレス不可）→ 将来 AWS/GCP。テスト CI。
