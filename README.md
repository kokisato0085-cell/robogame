# RoboGame（仮称）

ロボットの **パーツ** と **行動アルゴリズム** を組み立て、組み上げたロボット同士を自動で戦わせて観戦する対戦ゲーム。
プレイヤーは戦闘中は操作せず、事前に組んだ構成と行動ロジックだけで勝敗が決まる（プログラミングゲーム／オートバトラー系）。

> ステータス: **段階1 実装中** — 設計フェーズ完了、骨格を実装中。

## 開発ログ（技術的障壁 → 解決策）

### T1: 決定論シミュレーションのコア
- **障壁**：戦闘を 2D 平面に拡張するにあたり、**決定論（同じ入力 → 必ず同じ結果）を保ちつつ移動・距離を扱う**必要がある。浮動小数点は環境差で誤差が出るため使えない。
- **解決**：座標・速度・距離を**ミリ単位の整数**（固定小数、`PositionScale=1000`）で保持し、距離は**整数平方根 `isqrt`**（ニュートン法）、方向は整数除算で算出。乱数・時刻・map 反復順など非決定要素を排除。両者の行動は**ティック開始スナップショットから同時解決**。単体テスト `TestDeterminism` で「同入力→同リプレイ」を保証。
- 関連：`server/sim/`（`types.go` / `build.go` / `behavior.go` / `sim.go` / `sim_test.go`）

### T2: ビルド検証 ＋ ストア ＋ REST API
- **障壁**：複雑な入れ子のビルド（素体＋パーツ＋ルールセット）の妥当性検証（スロット/重量/案A電力/ルールの条件・行動が既知か）を**どこに置くか**。また検証ロジックが**決定論を壊さない**か。
- **解決**：ビルドの意味論を所有する **`sim.ValidateBuild`** に検証を集約し、API はその境界で呼ぶだけにした（関心の分離）。検証用の許可集合は `map` で持つが、**`Simulate` 内では使わない**ため決定論に影響しない。指名チャレンジ制（挑戦者 vs 相手 → 相手の受信箱へ記録）を REST で実装。
- 関連：`server/sim/validate.go` / `server/store/store.go` / `server/api/api.go` / `server/main.go`

### T3: ビルド作成UI（組立 ＋ ルールエディタ）＋ APIクライアント
- **障壁**：ルールエディタの状態管理（チャネル別・優先順位・複数条件AND・編集途中の取り消し）と、フロント↔サーバーの型整合。
- **解決**：**ロジック層（型・ビルド計算 `buildStats`）を UI から分離**し vitest で単体テスト（R7）。UI 状態は `ruleset` と「追加中条件 `pending`」の2つに集約して再描画。編集途中の誤操作救済として**追加中条件に個別の削除（✕）**を用意。Vite プロキシ＋usePolling で WSL 開発環境の差異を回避。
- 関連：`client/src/{types,data,buildStats,api,main}.ts` / `buildStats.test.ts`

### T4: 名簿・挑戦・Canvas観戦
- **障壁**：2D 戦闘の観戦描画（ミリ座標→Canvas変換、HP/シールド/熱の複数バー、攻撃ビーム・破壊表現）と、リプレイ再生の部品化・再再生。
- **解決**：描画を **`player.ts` に分離**（sim と描画の分離を維持）し、リプレイの再生に徹してロジックを持たせない。最後のリプレイを保持して「再再生」を実装。名簿は挑戦者セレクト＋各相手の「挑戦」で `POST /api/challenge` → 返却リプレイをそのまま再生。
- 関連：`client/src/player.ts` / `client/src/main.ts`

### 関心の分離（R6）
sim パッケージを責務で分割：型定義（types）／実効ステータス算出（build）／ルール評価（behavior）／戦闘ループ・幾何（sim）／検証（validate）。サーバーは API・ストア・sim に層分け。フロントは型／カタログ／計算／API／UI に分離。
