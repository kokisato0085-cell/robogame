# RoboGame（仮称）

ロボットの **パーツ** と **行動アルゴリズム** を組み立て、組み上げたロボット同士を自動で戦わせて観戦する対戦ゲーム。
プレイヤーは戦闘中は操作せず、事前に組んだ構成と行動ロジックだけで勝敗が決まる（プログラミングゲーム／オートバトラー系）。

## このゲームの狙い
- **パーツ選択と行動アルゴリズムを自由に組む**ことを追求したゲームシステムの確立。**2D でも面白い**ことを目指す。
- 本作の戦闘システムは、次回作 **Unity 製 3D 版 RoboGame のたたき台**。3D 化より先に、ゲームの根幹を 2D で作り切る。
- ※現状ロボの見た目は色付き円のプレースホルダ。スプライト画像への差し替えを予定（段階3 のUI刷新時）。

> ステータス: **段階3 完成（パーツ・機体・UI刷新）** — ダッシュ・防御・素体3種・UIタブ化・ロボsprite。次は段階4（リソース管理）。

## 開発ログ（技術的障壁 → 解決策）

### T1: 決定論シミュレーションのコア
- **障壁**：戦闘を 2D 平面に拡張するにあたり、**決定論（同じ入力 → 必ず同じ結果）を保ちつつ移動・距離を扱う**必要がある。浮動小数点は環境差で誤差が出るため使えない。
- **解決**：座標・速度・距離を**ミリ単位の整数**（固定小数、`PositionScale=1000`）で保持し、距離は**整数平方根 `isqrt`**（ニュートン法）、方向は整数除算で算出。乱数・時刻・map 反復順など非決定要素を排除。両者の行動は**ティック開始スナップショットから同時解決**。単体テスト `TestDeterminism` で「同入力→同リプレイ」を保証。
- 関連：`server/sim/`（`types.go` / `build.go` / `behavior.go` / `sim.go` / `sim_test.go`）

### T2: ビルド検証 ＋ ストア ＋ REST API
- **障壁**：複雑な入れ子のビルド（素体＋パーツ＋ルールセット）の妥当性検証（スロット/重量/案A電力/ルールの条件・行動が既知か）を**どこに置くか**。また検証ロジックが**決定論を壊さない**か。
- **解決**：ビルドの意味論を所有する **`sim.ValidateBuild`** に検証を集約し、API はその境界で呼ぶだけにした（関心の分離）。検証用の許可集合は `map` で持つが、**`Simulate` 内では使わない**ため決定論に影響しない。指名チャレンジ制（挑戦者 vs 相手 → 相手の受信箱へ記録）を REST で実装。
- 関連：`server/sim/validate.go` / `server/store/store.go` / `server/api/api.go` / `server/main.go`

### T3: ビルド作成UI（組立 ＋ ルールエディタ）＋ APIクライアント
- **障壁**：ルールエディタの状態管理（チャネル別・優先順位・複数条件AND・編集途中の取り消し）と、フロント↔サーバーの型整合。
- **解決**：**ロジック層（型・ビルド計算 `buildStats`）を UI から分離**し vitest で単体テスト（R7）。UI 状態は `ruleset` と「追加中条件 `pending`」の2つに集約して再描画。編集途中の誤操作救済として**追加中条件に個別の削除（✕）**を用意。Vite プロキシ＋usePolling で WSL 開発環境の差異を回避。
- 関連：`client/src/{types,data,buildStats,api,main}.ts` / `buildStats.test.ts`

### T4: 名簿・挑戦・Canvas観戦
- **障壁**：2D 戦闘の観戦描画（ミリ座標→Canvas変換、HP/シールド/熱の複数バー、攻撃ビーム・破壊表現）と、リプレイ再生の部品化・再再生。
- **解決**：描画を **`player.ts` に分離**（sim と描画の分離を維持）し、リプレイの再生に徹してロジックを持たせない。最後のリプレイを保持して「再再生」を実装。名簿は挑戦者セレクト＋各相手の「挑戦」で `POST /api/challenge` → 返却リプレイをそのまま再生。
- 関連：`client/src/player.ts` / `client/src/main.ts`

### T5: 受信箱
- **障壁**：非同期チャレンジ制では、挑まれた側（防衛側）が後から結果＋リプレイを確認する導線が必要。勝敗を防衛側視点で表す必要もある。
- **解決**：`GET /api/inbox?owner=` で防衛側の対戦を取得し、**防衛側視点に勝敗を変換**（winner=1→勝ち）。各結果から `player` を再利用してリプレイを観戦。これで段階1の背骨（登録→相手選択→挑戦→観戦→受信箱）が一気通貫。
- 関連：`client/src/main.ts` / `client/src/api.ts`

## 段階1 完成
非同期チャレンジ制の骨格が動作。次は中方針 中8 の段階2（エネルギー/案B・ダッシュ・装甲・複数素体＆個性など）で「アルゴリズム > パーツ」を本格成立させる。設計は [docs/POLICY.md](./docs/POLICY.md) を参照。

### S2-T1: 戦闘モデルの進化（発射体・遮蔽物・マップ）
- **障壁**：(1) hitscan（即命中）では長射程＋装甲ゴリ押しが最適解になりアルゴリズムが死ぬ。(2) 発射体を導入したら、弾速40 > 命中直径36 で**敵をすり抜ける（トンネリング）**。(3) 正面カバー＋直進approachで**ロボが詰む**。
- **解決**：(1) **発射体システム**（弾速・回避可・遮蔽物/射程で消滅）に置換し、回避・カバーで立ち回りが効くようにした。(2) 命中判定を点距離から**移動線分と円のスイープ判定** `segHitsCircle` に変更（整数・決定論）。(3) **斜め初期配置＋壁ずり移動**（軸別判定で遮蔽物を回り込む）。
- 関連：`server/sim/sim.go`（projectile/obstacle/slide/segHitsCircle）/ `types.go`
- ※対称配置の戦闘は相互遮蔽で引き分けもありうるため、テストは「弱い側は勝てない（負けない）」で検証。

### S2-T2: フロント描画＋膠着修正
- **障壁**：(1) 観戦をマップ拡大・遮蔽物・発射体に対応させる。(2) 指令なし（デフォルト接近）の2体が、**中央の障害物上で正面から釣り合って固まる**（対称な接近の均衡）。
- **解決**：(1) リプレイの `arenaW` を使った動的スケールで描画し、遮蔽物（灰矩形）と発射体（点）を追加。(2) **中央の接近ラインを空ける**レイアウトに変更（カバーは脇へ）＋壁ずりに**正面衝突時の垂直回り込み**を追加。回帰テスト `TestDefaultApproachReachesEnemy`（指令なしでも接近できる）で担保。
- 関連：`client/src/player.ts` / `client/src/types.ts` / `server/sim/types.go`（レイアウト）/ `sim.go`（壁ずり）
- 補足：現武器は射程250 < 初期距離≈1237 のため開幕は射程で攻撃が通らない。前方カバーは長射程武器を入れる段階3で再設置。

### 段階2 完成
発射体・遮蔽物・マップ拡大が一通り動作（S2-T1 sim ✅ / S2-T2 描画 ✅）。次は段階3（パーツ・機体・UI刷新）。

### S3-T1: パーツ拡張（ダッシュ・防御・素体差）の sim 実装
- **障壁**：ダッシュ・防御・複数素体を、決定論を保ったまま既存の戦闘ループへ整合的に足す。特に防御の被ダメ軽減を、**発射体が命中する側のタイミング**で正しく適用する必要がある。
- **解決**：防御は行動決定時に `Defending` フラグを立て、**命中処理 `applyDamage` で被ダメ ×1/2**＋移動量 ×1/4。ダッシュは移動 step を `dashDistance` に置換し `dashCd` を管理（`dashReady` 条件が有効化）。`derive`/`validate` に movement・defense カテゴリの認識を追加（素体は既存 derive が任意の値を扱える）。テスト先行で半減・CD・検証を担保。
- 関連：`server/sim/sim.go` / `build.go` / `validate.go` / `types.go`

### S3-T2: ビルダー汎用化＋戦闘の可読化＋バランス調整
- **障壁**：(1) 単一武器トグルを、素体3種選択＋パーツカタログから複数装着できる汎用ビルダーへ。(2) 戦闘が読みにくい（ダメージ・防御が見えない）。(3) ガードが強すぎ＆重い機体が軽い機体に押された。(4) ガード表示が「防御姿勢」基準で、耐久0でも残る／半減してないのに G 表記。
- **解決**：(1) 素体セレクタ＋パーツカタログ（チェックで装着・制約リアルタイム表示）＋行動パレット拡張（dash/defend/dashReady）。(2) **ダメージ数値の浮遊表示**＋**ガードの分割リング**（被弾で欠ける）。(3) **ガードに耐久値5**（5回まで半減・使い切ると無効）＋**押し合いを重量比で配分**（重い方は押されにくい・速さで詰める力は維持）。(4) 攻撃イベントに「実際に半減したか `guarded`」を持たせ、リングは `defending && 残量>0` の時だけ表示。
- 関連：`client/src/{data,main,player,types}.ts` / `server/sim/{types,build,validate,sim}.go`

### S3-T3: UIタブ化＋owner記憶
- **障壁**：1ページに全部が並んで雑然としていた。所有者名を毎回入力するのが手間。
- **解決**：**3タブ**（ロボ作成／対戦／受信箱）に再編し、対戦・受信箱は各タブ専用の観戦Canvas＋player を持つ。**owner名を localStorage に記憶**し、リロード後も自動入力（本格アカウントは段階5）。
- 関連：`client/index.html` / `client/src/main.ts`

### S3-T4: ロボ sprite 描画
- **障壁**：色付き円では味気ない。トップダウン視点でどう「向き」を見せるか。配布画像が 2048px・6.8MB と重く、公開リポジトリを肥大化させたくない。
- **解決**：正面／後ろ向きの2枚を**ビルボード**として表示し、**敵が下にいれば正面・上にいれば後ろ向き**で出し分け（向きが伝わる）。画像は **PIL で 256px（約110KB）に縮小**してからコミット（履歴を軽く保つ）。観戦画面を 760px に拡大。陣営の青/赤はバー色で区別。
- 関連：`client/src/player.ts` / `client/public/robot-{front,back}.png`

### 関心の分離（R6）
sim パッケージを責務で分割：型定義（types）／実効ステータス算出（build）／ルール評価（behavior）／戦闘ループ・幾何（sim）／検証（validate）。サーバーは API・ストア・sim に層分け。フロントは型／カタログ／計算／API／UI に分離。
